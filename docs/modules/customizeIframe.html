customizeIframe

Injects a <style> element into an iframe's document, letting you override third-party styles without touching the host page. Every rule is automatically suffixed with !important, so your overrides win even against deeply-nested vendor defaults. Useful when a service (e.g. HubSpot, Typeform, Intercom) locks visual customization behind a paid plan.

Signature

customizeIframe(iframeId, styleElementId)

Parameters

Name Type Description
iframeId string The id of the target <iframe> element.
styleElementId string The id of a <style> or <template> element on the host page whose text content will be injected.

Usage

Define your override styles in a <style> tag on the host page, give it an id, then call customizeIframe with both ids. The function will wait for the iframe to appear in the DOM and finish loading before injecting — no manual timing required. 

<!-- Your override styles, kept out of the iframe -->
<style id="hubspot-overrides">
  .hs-button {
    background-color: #000;
    border-radius: 4px;
    font-family: inherit;
  }
</style>

<!-- The iframe (may be injected dynamically by a third-party script) -->
<iframe id="hubspot-form"></iframe>

<script src="https://cdn.marti.dev/customizeIframe.js"></script>
<script>
  customizeIframe('hubspot-form', 'hubspot-overrides');
</script>

The injected stylesheet becomes a normal <style> element inside the iframe's <head>. All declarations are rewritten to include !important automatically, so you don't need to add it manually in your source.

How it works

Because third-party embeds are often injected into the DOM after your script runs, a plain load event listener on the iframe is unreliable — the element may not exist yet. customizeIframe works around this by attaching a MutationObserver to document.body that watches for the iframe to appear. Once found, the observer disconnects itself and either injects immediately (if the iframe is already loaded) or waits for its load event first.

Limitations

Injection requires that the iframe and the host page share the same origin. Cross-origin iframes (e.g. an embed served from a different domain) block contentDocument access as a browser security measure — this function will have no effect on them. Whether a given third-party embed is same-origin depends on how the vendor serves it; some (like HubSpot's inline form embeds) do qualify.
